# MVP 收口周计划

目标：

在不调整现有源码目录结构的前提下，完成一轮面向可用性和推进效率的收口工作。

为什么现在做：

- 当前 MVP 主链路已经打通，下一阶段更需要提升错误可见性和交互完整度
- 这些工作能直接降低后续调试成本，并为继续补功能提供更稳定的基础
- 相比预览元素选中，这些任务范围更清晰，更适合当前阶段推进

范围：

- 补齐 AI 生成错误的前后端可见性
- 用正式导出弹窗替换当前基础导出交互
- 增加快速模式 / 高质量模式
- 做测试、构建和文档收口
- 不实现预览元素选中能力
- 不做物理目录迁移

步骤：
1. 梳理生成链路里的错误来源、错误分级和展示策略，并更新相关任务说明或 README
2. 实现详细错误可见性，补齐前端展示、后端错误映射和对应测试
3. 实现正式导出弹窗，明确导出类型、目标位置、命名规则以及成功失败反馈
4. 增加快速模式 / 高质量模式，打通前后端参数传递与设置持久化
5. 完成回归验证，更新 `known-gaps`、`backlog` 和执行中任务内容

当前进度：

- 步骤 1 已完成
- 步骤 2 已完成
- 步骤 3 已完成
- 步骤 4 已完成
- 步骤 5 已完成：测试、构建和文档已同步

验收标准：

- 生成失败时，用户能看到明确原因和下一步建议，而不是笼统报错
- 导出流程通过正式弹窗完成，且具备基础成功失败反馈
- 生成模式可切换，且前后端链路真实生效
- `npm test -- --run`、`cargo test --lib`、`npm run build` 保持通过
- 文档中的当前状态、缺口和任务内容与代码一致
