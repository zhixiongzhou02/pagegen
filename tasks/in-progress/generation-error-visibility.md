# 生成错误可见性任务

当前状态：

- 已完成前端结构化错误展示，错误信息支持标题、详情和建议
- 已完成 IPC 层错误解包，前端可稳定拿到后端错误类型和消息
- 已完成常见生成失败映射，包括超时、API Key、模型不可用、非法 HTML 和服务异常
- 已补前端测试，覆盖错误格式化、IPC 错误透传和页面展示
- 该任务可视为当前阶段完成，后续只在发现新错误模式时增补映射

目标：

把页面生成失败从“只有一条笼统错误字符串”提升为“有分类、有原因、有建议的可读反馈”。

为什么现在做：

- 页面生成是核心链路，当前错误反馈过于笼统，直接影响可用性和排障效率
- 生成模式切换和导出交互优化都依赖更清晰的错误出口
- 这项工作范围清晰，适合作为本周第一个收口任务

范围：

- 梳理前端、IPC、Rust 后端的错误传递链路
- 建立生成错误分类和用户可读文案
- 调整前端状态与展示，支持错误标题、详情和建议
- 为错误映射与展示补测试
- 不改动生成算法本身
- 不引入复杂的全局消息系统

步骤：
1. 盘点当前生成链路中的错误来源，包括设置缺失、请求超时、提供方错误、返回内容非法、项目或页面不存在
2. 设计统一错误结构，至少包含分类、短标题、详情和建议，并确定前后端各自负责的部分
3. 更新 Rust 错误出口和 IPC 映射逻辑，避免直接把内部报错原样暴露给 UI
4. 更新前端状态与展示，让生成失败信息支持更清晰的结构化呈现
5. 为错误格式化、IPC 映射和界面展示补测试，并完成回归验证

验收标准：

- 用户在生成失败时能看到明确失败原因
- 常见错误至少覆盖 API Key 缺失、请求超时、模型不可用、返回内容非法、网络或服务异常
- 错误提示包含下一步建议，而不是只重复底层异常
- 不破坏现有生成成功流程
- 相关前端和后端测试保持通过

当前代码观察：

- 前端当前主要通过 `formatUserFacingError(action, message)` 返回单一字符串
- IPC 层当前只提取 `message` 或 `type`，没有保留结构化错误信息
- Rust 侧 `PageGenError` 已有可序列化枚举，但命名仍偏底层，且缺少更贴近生成场景的分类
- 应用状态里目前只有 `error: string | null`，不利于展示详情和建议

建议拆分：

- 前端
  - 将错误状态从单一字符串升级为可选结构化对象
  - 为错误展示区域增加标题、详情、建议三段式信息
  - 保留普通流程的成功提示和加载状态
- IPC
  - 保留后端传回的 `type` 与 `message`
  - 在前端统一转换成用户态错误对象
- Rust
  - 明确 AI 生成场景下的错误类别
  - 对第三方服务、超时、无效响应等场景给出稳定错误消息
- 测试
  - 补 `utils` 或新错误映射模块测试
  - 补 `ipc` 测试
  - 补 `App` 中生成失败展示测试
