# 源码结构治理策略

这个文档定义当前阶段的源码结构原则。核心约束是：不调整现有源码目录结构，不做物理搬迁式重构。

## 当前决策

- 保持 `src/` 和 `src-tauri/` 现有目录结构不变
- 不为了“看起来更合理”而移动源码文件
- 后续优化以文档治理、流程治理、局部收口为主
- 只有在部署、构建、使用稳定不受影响的前提下，才允许非常小范围的本地整理

## 为什么这样做

- 当前项目仍在持续补功能和修稳定性
- 大规模目录重构会直接增加 import、构建、测试、配置的风险
- 当前更缺的是治理能力，而不是目录层级本身
- 现阶段应该优先降低推进成本，而不是追求结构美观

## 当前问题怎么解决

不通过改目录来解决，而通过下面手段控制复杂度：

1. 用分层 README 控制阅读路径
2. 用 `tasks/` 管理任务状态
3. 用 `planning / tdd / definition-of-done` 控制执行流程
4. 用 `ADR` 沉淀关键设计原因
5. 用 `Error Policy` 和 `Performance Budget` 控制运行质量
6. 用局部提炼代替目录级搬迁

## 允许的优化

- 拆小过长函数
- 提炼 helper
- 增加测试
- 增加目录级 README
- 增加任务与决策文档
- 在不改文件路径的前提下优化模块边界

## 暂不允许的优化

- 新建大规模功能分层目录并迁移源码
- 修改 `src/` 下现有文件的大量路径
- 修改 `src-tauri/src/` 下模块布局
- 为了“更规范”而一次性重排目录

## 后续优先级

1. 提高错误可见性
2. 优化 AI 生成耗时
3. 增强预览与编辑联动
4. 明确模块边界文档
5. 只在必要时做小范围本地整理

## 重新评估条件

只有满足下面条件，才重新考虑源码目录重构：

- 核心工作流稳定
- 主要功能不再快速变化
- 现有目录已经明显阻碍开发效率
- 有清晰收益，且不会影响部署与使用
